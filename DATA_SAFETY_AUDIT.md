# 🛡️ 데이터 안전 시스템 전체 감사 (Data Safety Audit)

> 작성일: **2026-06-13** · 대상: `index.html` (패션 생산관리 PRO)
> 목적: 백업·동기화·자동저장·복원 시스템을 한눈에 점검하고, 발견된 허점과 수정 방향을 기록한다.
> 계기: 6/13 데이터 유실 사고(쿠키 삭제 + 핸드폰 동기화로 이미지·오더 손실) 진단.

## 0. 한 줄 요약

데이터는 **5곳**에 저장된다. 브라우저 안(2곳)은 쿠키 삭제에 취약하고, 클라우드·파일(3곳)은 안전하다.
**이번 사고의 핵심 원인은 2개**: ① 쿠키 삭제로 브라우저 안(localStorage+IndexedDB)이 통째로 지워짐 ② 핸드폰의 불완전한 데이터가 클라우드의 완전한 데이터를 덮어씀(안전장치 허점).

## 1. 데이터가 사는 곳 (저장소 5곳)

| # | 저장소 | 무엇이 들어있나 | 쿠키 삭제 | 멀티기기 | 한도 |
|---|--------|----------------|:--------:|:--------:|------|
| 1 | **localStorage** (브라우저 안) | 실데이터 주 저장소 (아이템·오더·단가장 등) | ❌ **삭제됨** | 기기별 별도 | ~5MB |
| 2 | **IndexedDB** (브라우저 안) | 자동백업 (저장백업 5개 + 매시간 12개) | ❌ **삭제됨** | 기기별 별도 | 큼 |
| 3 | **Firebase Firestore** (클라우드) | 동기화/백업 (회사별 전체 데이터) | ✅ 안전 | **공유** | 1MB/문서 |
| 4 | **Firebase Storage** (클라우드) | 작업지시서 **이미지 파일** | ✅ 안전 | **공유** | 큼 |
| 5 | **다운로드 폴더 파일** (완전 밖) | 수동/자동 백업 JSON | ✅ 안전 | 기기별 | 무제한 |

> 💡 **핵심 원리**: 화면에 데이터를 빠르게 띄우려고 localStorage(브라우저 안)를 "작업용 사본"으로 쓰고, 클라우드·파일을 "안전 보관"으로 쓴다. 작업용 사본(브라우저 안)은 브라우저 청소로 지워질 수 있다.

## 2. 백업 시스템 (4종)

| 백업 | 함수 | 저장 위치 | 시점 | 보관 개수 | 빈데이터 가드 |
|------|------|----------|------|:--------:|:------------:|
| **저장 백업** | `autoLocalBackup` | IndexedDB (`fpm_abk_*`) | 저장 시 (90초 간격/개수 변경) | 5개 | ✅ 있음 |
| **매시간 백업** | `autoHourlyBackup` | IndexedDB (`fpm_hbk_*`) | 매시간 (새벽 0~6시 제외) | 12개 | ✅ 있음 |
| **자동 파일백업** | `autoFileBackup` | 다운로드 폴더 | 하루 2회 (14시·21시) | 무제한 | ✅ 있음 (2026-06-13 추가) |
| **수동 백업** | `exportData` | 다운로드 폴더 | 사용자가 버튼 클릭 | — | 없음(수동이라 무관) |

- **저장 백업/매시간 백업**은 IndexedDB에 있어 **쿠키 삭제 시 함께 사라진다** → 정작 필요할 때 비어있을 수 있음 (이번 사고).
- **자동 파일백업**(신규)은 다운로드 폴더라 쿠키 삭제와 무관 → **현재 가장 믿을 만한 안전망**.
- 빈데이터 가드: 아이템·오더·브랜드가 모두 0개면 백업하지 않음 (사고 시 빈 백업이 쌓이는 것 방지).

## 3. 자동 저장 흐름 (`saveData`)

저장 버튼이 아니라, 데이터 변경 시 자동 호출. 호출되면 **5곳에 순서대로** 반영:

```
saveData()
 ├─ autoLocalBackup()      → IndexedDB 저장백업 (빈데이터면 skip)
 ├─ autoHourlyBackup()     → IndexedDB 매시간백업 (빈데이터면 skip)
 ├─ lsSet('fpm_<회사>')     → localStorage 실데이터  ← 실패 시 _onSaveFailed (백업파일 자동 다운로드)
 ├─ lsSet('fpm_shared')    → localStorage 공장·단가장
 └─ cloudSyncSave()        → Firestore (800ms 디바운스, 빈데이터 가드)
```

- **저장 실패 방어**: localStorage가 꽉 차거나(용량 초과) 시크릿 모드면 `_onSaveFailed`가 작동 → 경고 + 백업파일 자동 다운로드 (20초에 1회).
- **자동 파일백업**은 saveData가 아니라 별도 타이머(1분마다 시각 확인)로 작동.

## 4. 동기화 시스템 (클라우드 ↔ 기기)

| 함수 | 방향 | 동작 |
|------|------|------|
| `cloudSyncSave` | 기기 → 클라우드 | 저장 시 클라우드에 푸시. **dataURL 사진 제외**(1MB 한도). 빈데이터 가드. |
| `cloudSyncLoad` | 클라우드 → 기기 | 클라우드에서 받음. **자동 로드 = 로컬 우선**(로컬에 데이터 있으면 안 받음). 강제(force)만 덮어씀. |
| `startRealtimeSync` | 클라우드 → 기기 (실시간) | `onSnapshot`으로 다른 기기 변경을 즉시 반영 (`applyRemoteCoData`). |
| `onFirebaseLogin` | 오케스트레이터 | 로그인 시: 클라우드 회사목록 → loadCoData → cloudSyncLoad → startRealtimeSync. |

### 이미지 동기화 (작업지시서)
- 이미지는 **Firebase Storage에 파일로** 올라가고, 데이터에는 **URL만** 저장 (1MB 문서 한도 회피).
- `tpStripDataUrls`: 클라우드 저장 시 아직 업로드 안 된 dataURL 사진은 제외 (URL만 보존).
- `tpMergeLocalImages`: 원격 데이터 받을 때 로컬 **dataURL** 사진을 다시 합침.
- `tpPreserveLocalContent`: 원격 적용 시 로컬 보호 (원단·부자재·색상 있는 아이템, 공장 배정).

## 6. 🔴 발견된 허점 (이번 사고 원인 + 위험)

### 허점 ① — 쿠키 삭제 = 브라우저 안 전멸 [치명·구조적]
- localStorage(실데이터) + IndexedDB(자동백업) 둘 다 "사이트 데이터"라 크롬 쿠키 삭제로 **동시에** 지워진다.
- **방어**: 클라우드 연동 + 자동 파일백업(둘 다 브라우저 밖). 크롬 설정에서 사이트 예외 등록.

### 허점 ② — 클라우드 덮어쓰기 안전장치가 "완전 빈 경우"만 막음 [치명·이번 사고 직접원인]
- `cloudSyncSave`(line 2095): 아이템·오더·브랜드가 **전부 0개**일 때만 클라우드 덮어쓰기를 막음.
- **부분 손실은 통과**: 이번엔 아이템 71개는 그대로고 이미지 28→0, 오더 40→34만 줄었는데, "비어있지 않다"고 판단해 통과 → 클라우드를 손상된 데이터로 덮어씀.
- **수정 방향**: 클라우드보다 이미지/오더/아이템 수가 크게 줄면 → 덮어쓰기 전 경고·차단.

### 허점 ③ — `applyRemoteCoData`가 오더·브랜드를 무조건 덮어씀 [높음]
- line 1911: items만 `tpPreserveLocalContent`로 보호하고, **orders·brands·sampleOrders는 원격 값으로 그냥 교체**.
- 다른 기기가 빈/적은 오더로 푸시하면 → 내 오더가 그 즉시 줄어듦 (오더 40→34 메커니즘).
- **수정 방향**: orders 등도 "원격이 더 적으면" 경계하거나 개수 가드 적용.

### 허점 ④ — 작업지시서 URL 이미지가 동기화에서 보존 안 됨 [높음]
- `tpMergeLocalImages`는 **dataURL만** 다시 합치고, **Storage URL 이미지는 보존 대상이 아님**.
- 원격이 techpack 이미지를 비운 채 보내면 → 로컬 URL 이미지도 사라짐 (이미지 28→0 메커니즘).
- **수정 방향**: URL 이미지도 보존 대상에 포함.

### 허점 ⑤ — `cloudSyncLoad` 로컬 우선 → 오래된 기기가 최신을 안 받음 [중간]
- 자동 로드 때 로컬에 데이터가 있으면 클라우드 최신을 받지 않음(수정 보호 목적).
- 부작용: 핸드폰처럼 **옛 데이터를 가진 기기가 최신을 못 받고, 나중에 그 옛 데이터로 클라우드를 덮어씀.**
- **수정 방향**: 타임스탬프 비교 또는 "다른 기기가 더 최신이면 받기" 안내.

## 7. 🟢 잘 되어 있는 것 (믿어도 되는 부분)

- ✅ **실시간 동기화**(`startRealtimeSync`): onSnapshot + 편집 중 보호 배너 + 내 쓰기 무시 처리 — 견고함.
- ✅ **저장 실패 방어**(`_onSaveFailed`): 용량 초과/시크릿 모드 시 자동 백업파일 다운로드.
- ✅ **빈데이터 백업 방지**: 자동백업 3종 모두 빈 상태면 skip.
- ✅ **파일 복원 시 클라우드 보호**(`fpm_force_local`).
- ✅ **이미지 = Storage 파일**: 쿠키 삭제해도 이미지 원본은 구글 서버에 안전 (이번에 확인됨).
- ✅ **자동 파일백업**(신규): 하루 2회 다운로드 폴더에 — 브라우저 청소와 무관한 안전망.

## 9. 사용자 행동 수칙 (재발 방지)

1. **크롬에서 쿠키/사이트 데이터/방문기록 삭제 시 → 이 앱 사이트(`subak805-netizen.github.io`) 제외.**
2. **크롬 설정 → "항상 데이터를 저장할 수 있는 사이트"에 앱 주소 등록.**
3. **여러 기기(PC·핸드폰)에서 쓸 때**: 한 기기에서 작업 후, 다른 기기는 열 때 **☁ 새로고침**으로 최신을 먼저 받기. (허점 ⑤ 수정 전까지)
4. 자동 파일백업(하루 2회)이 쌓이는지 가끔 다운로드 폴더 확인.
